# プールアロケータの設計（調査フェーズ）

## 目的

現在、RISC-V ランタイム（`bootstrap/runtime_syscall.c`）は bump allocator で
heap を消費するだけの実装になっており、解放できない。そのため、コンパイラ
自己ホスト時のメモリ消費が 9GB 超に達し、32MB の bump heap を使い切って
OOM になる。これを回避するため `heap_mark`/`heap_reset` を追加したが、
ユーザーの方針としてはこのスコープベース回収を廃止し、**free できる
malloc**（サイズ倍々のプールアロケータ）を導入したい。

本ドキュメントは、そのプール設計に必要な **現状の割り当てサイズ分布**
を実測して整理したもの。

## 計測方法

`bootstrap/bcrun.c` に一時的にインストルメンテーションを仕込み、以下を
記録した（実験後 revert 済み）:

- `XxxArray(sz)` が要求した backing buffer サイズ（`sz * 4` バイト）
- `append(String, …)` / `append(String, String)` の bytes バッファ長
- `delete(obj)` 呼び出し
- `heap_mark()` / `heap_reset(mark)` 対による一括 free

backing buffer のバイト数を bucket 化（power-of-2）し、**累計 alloc 回数**
と **同時 live 数のピーク** を記録。文字列リテラル（`push_str` 由来）は
rodata を指すだけなので対象外。`HeapObj` ヘッダ（固定 ~32B）も対象外
（1 pool で十分なため）。

## 計測ワークロード

いずれも bcrun 上で自己ホストコンパイラの 1 ステージを動作させたもの。
入力は parse.tc など大きめの .tc ソース 1 本を想定。

| ステージ         | 入力                       | 備考                                    |
|------------------|----------------------------|-----------------------------------------|
| parse.tc         | parse.tc (60KB, 1638 行)    | 最も alloc が多い。heap_mark/reset を使用 |
| typecheck.tc     | codegen.tc の AST           | heap_mark/reset 未使用                   |
| codegen.tc       | parse.tc の TAST            | heap_mark/reset 未使用                   |
| bc2asm.tc        | parse.tc の BC              | heap_mark/reset 未使用                   |

## 知見

1. **小さい alloc が回数を支配**。`size <= 128` が全 alloc 回数の大半。
   特に `8`（struct header）と `48`（ob_int の tmp）、`96`（引数スタック）
   の 3 サイズが圧倒的に多い。
2. **大きな alloc が累計バイトを支配**。parse.tc の scratch
   `U8Array(65536)` = 262144B が 3.6 万回繰り返され throughput のほぼ
   全て。ただし同時 live は 34 個（再帰降下の depth に比例）。
3. **中間サイズ（256B〜32KB）は alloc 回数が少ない**。シンボル表やソース
   バッファなど、プログラム全体で 1〜3 個しかないヘビーオブジェクト。
4. **どのステージも peak live は ~15MB 以下**で、free して回せば 32MB
   の物理ヒープで十分余裕がある。

## プール設計案

### サイズクラス

power-of-2 の倍々で定義。64B 未満は細かめ、64B 以上は倍々:

```
  16,  32,  64,  128,  256,  512,  1024,  2048,  4096,
  8192, 16384, 32768, 65536, 131072, 262144, 524288,
  1048576, 2097152, 4194304
```

要求サイズを切り上げて該当バケットに割り当てる。

### 各バケットの必要数（peak live の最大値 × 各ステージで）

4 ステージの中で最大の peak live 数:

| bucket (B) | 必要スロット数 | 確保バイト数    |
|-----------:|---------------:|----------------:|
|         16 |             35 |             560 |
|         32 |              3 |              96 |
|         64 |              2 |             128 |
|        128 |              4 |             512 |
|        256 |              1 |             256 |
|       1024 |              — |               — |
|       2048 |              1 |           2,048 |
|       4096 |              1 |           4,096 |
|       8192 |              1 |           8,192 |
|      16384 |              3 |          49,152 |
|      32768 |              — |               — |
|      65536 |              2 |         131,072 |
|     131072 |              1 |         131,072 |
|     262144 |             34 |       8,912,896 |
|     524288 |              2 |       1,048,576 |
|    1048576 |              2 |       2,097,152 |
|    2097152 |              2 |       4,194,304 |
|    4194304 |              1 |       4,194,304 |

合計 peak 必要量: **約 20.8 MB**（backing buffer のみ）。
32MB の RISC-V ヒープに余裕で収まる。

### 実装方針

各 bucket に対し:

- **初期化時** に固定サイズのスラブを 1 本確保（遅延でも可）。
- **free list** をリンクリストで管理。スラブ内の各スロットの先頭
  `sizeof(void*)` バイトを次ポインタとして流用。
- `alloc(size)`: size を bucket に切り上げ、free list から 1 つ pop。
  空ならスラブを追加確保（bump 的に拡張）。
- `free(ptr)`: alloc 時に記録した bucket id を使って free list に push。
  bucket id の保持方法は 2 案:
  1. `ptr` の直前にヘッダ（1 word）を置いて bucket id を記録。
  2. スラブごとに固定 bucket id なので、`ptr` のアドレス範囲から
     bucket id を引ける表を持つ（page-level round down）。
- `delete(HeapObj*)` がこの `free()` を呼ぶ。
- `heap_mark`/`heap_reset` は **廃止**（または単純に no-op 化）。

### 注意

- 262144B バケットだけで 8.9MB を占める。parse.tc のスクラッチを
  65536 要素より小さく作れれば大きく削減可能（ただし大きな式を
  含むソースでは overflow する可能性があるので要検討）。
- `append_char` のインクリメンタルな文字列構築は長い文字列を作るとき
  に O(n^2) で alloc が発生するため、出来れば `OutputBuffer` に置き
  換えるのが望ましい。

## 参考

計測コード差分（revert 済み）は `bootstrap/bcrun.c` への一時パッチで
再現可能。主要な計測点は `heap_alloc` 直後の `stats_record_alloc()` と
`delete` builtin / `heap_reset` builtin での `stats_record_free()`。
